2. Problem Formulation
The SLM deployment decision operates at three distinct levels. Conflating them is the principal source of miscommunication between the runtime routing literature and the technology management literature.
2.1 Enterprise Task (Use-Case Level)
An enterprise task is the business-process unit for which a deployment decision is made before any inference is executed. It has a defined input type, required output format, volume profile, latency budget, and risk class. Eight tasks are studied here, spanning six industry domains. Each is characterized by a 100-item gold set partitioned 70/30 into development and test splits, with difficulty stratified across easy, medium, and hard items. All dimension scores and tier predictions were locked in the gold-set metadata prior to any model inference (inference_config.json, locked_date: 2026-03-02).
2.2 NLP Task Family (Task-Family Level)
An NLP task family is the computational abstraction at which SDDF v3 operates. Enterprise tasks that share the same computational structure map to the same family. The mapping is computed by the s3_sddf_bridge.py script from the outputs of both repositories and is shown in Table 1.

2.3 Optimization Objective
The deployment objective is to select, for each enterprise task δ, the tier T(δ) that serves queries locally (Pure SLM) wherever the SLM is capable and safe, and escalates to the LLM baseline otherwise. Formally:
T*(δ) = argmin_{T} Cost(T)   subject to   C_m(d) ≥ θ_cap(d),   R_m(d) ≤ θ_risk(d)
where C_m(d) is the per-model capability at difficulty level d, R_m(d) is the expected failure-severity risk at difficulty d, and θ_cap(d) and θ_risk(d) are adaptive thresholds computed from the validation split. This objective is operationalized in SDDF v3 as follows (sddf/validation_dynamic.py):
Per-sample capability is 1 if the SLM output is correct, 0 otherwise.
Per-sample risk is 0 on correct outputs; on failures it is task_risk_base × severity_multiplier × category_multiplier, where task_risk_base is task-specific (e.g. 0.25 for summarization, 0.57 for code generation, 0.90 for mathematics).
Capability and risk curves are estimated over difficulty bins using isotonic regression (non-decreasing for C_m, non-increasing for R_m).
τ* is selected as the maximum difficulty score satisfying both constraints simultaneously. When no feasible region exists, a minimum-violation fallback selects the bin with the lowest combined constraint violation (tau_source = ‘fallback_min_violation’).
Thresholds θ_cap and θ_risk are set adaptively as percentiles of the empirical capability and risk distributions over the validation split (default: 50th percentile for capability, 75th for risk), rather than fixed constants.
S³ approximates this objective through expert-scored dimensions before any inference. SDDF v3 solves it empirically from benchmark data. The two frameworks are compared in Section 8.
3. Research Methodology
3.1 Design Science Paradigm
We follow the design science research paradigm, which positions the artefact as the primary contribution and requires demonstration and evaluation against relevance and rigour criteria.
Relevance: enterprises lack an auditable pre-deployment protocol for SLM tier assignment.
Rigour: S³ is grounded in weighted sum MCDM theory; SDDF v3 uses logistic regression failure modelling with isotonic calibration. Both are validated across eight use cases with pre-registered hypotheses.
3.2 Two-Artefact Design
S³: S³ (sddf/s3_framework.py): pre-deployment expert scoring. Implemented as compute_s3_score(), prescreen_gate(), and tier_from_s3(). The constraint w_SK ≥ w_TC is enforced at call time. Scores committed before benchmarking.
SDDF v3: SDDF v3 (framework/benchmarking/): empirical post-deployment validation. Three Python pipeline files handle train, validation, and test phases. Difficulty features extracted by sddf/difficulty.py (20 features per input). Data splits are deterministic (seed: slm-clean-v1, 60/20/20 ratio, confirmed in clean_deterministic_splits/summary.json).
3.3 Pre-Registration and Experimental Controls
All S³ dimension scores locked before inference (inference_config.json, locked_date: 2026-03-02). No scores revised after benchmarking.
Inference parameters fixed: temperature 0.0, top_p 1.0, seeds [42, 43, 44], three runs per item.
SDDF v3 uses seeds [42–46] for train-phase stability. Five sensitivity configurations vary validation hyperparameters.
Cross-framework correlation computed by s3_sddf_bridge.py from pre-existing outputs of both repositories; no new inference generated.
3.4 Research Questions
RQ1 (Tier Prediction): does S³ correctly predict the deployment tier for each enterprise use case?
RQ2 (Gate Rule Necessity): do non-compensatory gate rules prevent unsafe assignments that the weighted sum alone would produce?
RQ3 (Cross-Framework Convergence): do S³ scores correlate with SDDF routing capability across NLP task families?
RQ4 (Robustness): are tier assignments stable under perturbations of the weight vector across five organizational profiles?
