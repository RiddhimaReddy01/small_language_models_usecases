Abstract 
Enterprise adoption of generative artificial intelligence presents a structural decision problem for technology managers: 
determining which workloads are suitable for locally served small language models (SLM)  and which require escalation 
to cloud-hosted large language model (LLM) inference. Current practice relies on vendor benchmarks or managerial 
intuition, neither of which produces an auditable, reproducible, pre-deployment decision protocol. This paper introduces 
two complementary frameworks to resolve this constraint. The SВі framework (Suitability Score for SLM Selection) is a 
multicriteria decision-making instrument that scores enterprise workloads on six dimensions вҖ” Task Complexity, 
Output Sensitivity, Specialized Knowledge Requirement, Data Sensitivity, Latency Tolerance, and Volume Load вҖ” 
using an ablation-selected weighted sum model augmented with non-compensatory prescreening gate rules, mapping 
each workload to one of three deployment tiers: Pure SLM, Hybrid, or LLM Only. 
The SLM Deployment Decision Framework (SDDF, version 3) serves as the empirical validation layer, operating at the 
NLP task-family level through a three-phase train-validate-test protocol that estimates per-model capability curves C(d) 
and risk surfaces R(d) across difficulty levels, selects an optimal routing threshold П„*, and evaluates frozen artifacts on 
held-out inputs. The unified framework is validated across eight enterprise use cases spanning cybersecurity, finance, 
healthcare, legal, and customer service domains, benchmarking three Qwen SLMs (0.5BвҖ“7B parameters) against a 
Llama 3.3 70B cloud baseline. Cross-framework validation yields a Spearman rank correlation of вҲ’0.726 (p = 0.010) 
between expert-assigned SВі scores and data-driven routing capability, confirming convergent validity between top-down 
expert assessment and bottom-up operational telemetry. The complete experiment is fully reproducible on commodity 
hardware without a discrete GPU.

4. The SВі Framework  
SВі is a pre-deployment multicriteria decision-making framework that maps an enterprise workload to one of three 
deployment tiers before any inference infrastructure is provisioned. The framework consists of four components: a 
manager input protocol, a six-dimension scoring rubric, an ablation- selected weighted sum formula with a normalization 
denominator, and a set of prescreening gate rules that enforce non-compensatory safety constraints  
4.1 SВі Dimensions (Manager Inputs)  
A domain expert and a compliance reviewer jointly score the target workload using the anchored rubric in Table 2. For 
each use case u, six dimensions are scored on a 1вҖ“5 integer scale: 
вҖў TC: Task Complexity  
вҖў OS: Output Sensitivity  
вҖў SK: Specialized Knowledge Requirement  
вҖў DS: Data Sensitivity 
вҖў LT: Latency Tolerance (higher score = stricter latency constraint) 
вҖў VL: Volume Load (higher score = larger expected scale)  
 
The domain expert provides ground-truth knowledge of the workload's technical characteristics вҖ” the realistic range of 
inputs, the acceptable output format, the volume profile, and the latency budget. The compliance reviewer independently 
assesses the SK and DS dimensions, which require legal and regulatory expertise that domain experts may lack. 
Disagreements between reviewers are resolved through a structured adjudication round before scores are committed.

1.1 The SВі Formula (Ablation-Selected) 
Using ablation-selected weights, the manager score is computed as: 
 
рқ‘Ҷ3 = (3 TC + 2 OS + 5 SK + 2 DS + 2 
LT + 1 VL) / 15 
 
 
 
пӮ· Numerator: weighted sum of dimension scores 
пӮ· Denominator 15: sum of weights (3 + 2 + 5 + 2 + 2 + 1), ensuring рқ‘Ҷ3  remains on a 1вҖ“5 scale 
пӮ· The weight vector reflects an ablation-optimized enterprise risk profile: Specialized Knowledge Requirement 
(SK) receives the highest weight of 5, reflecting the empirical finding that tasks requiring domain-licensed or 
regulated expert knowledge are the most reliable predictor of SLM failure 
пӮ· enterprise settings. Task Complexity (TC) is weighted at 3, capturing the structural difficulty of the reasoning 
required. Output Sensitivity (OS), Data Sensitivity (DS), and Latency Tolerance (LT) are weighted equally at 
2, contributing moderate and balanced influence. Volume Load (VL) receives the minimum weight of 1 as a 
tiebreaker вҖ” while high volume amplifies the cost savings from local deployment, it does not independently 
determine whether local deployment is safe or capable. The constraint w_SK вүҘ w_TC is enforced at runtime 
in compute_s3_score() to prevent any custom weight profile from inadvertently prioritizing capability over 
safety. 
пӮ· A lower SВі score indicates higher suitability for local SLM deployment; a higher score indicates the workload 
demands LLM capability, oversight, or both.

1.2 Prescreening Gate Rules 
Weighted sum models allow high scores on some dimensions to compensate for low scores on others 
вҖ” a property known as full compensability. In safety-sensitive enterprise settings, full compensability is 
undesirable: a workload with catastrophic failure consequences should not be routed to a local SLM merely 
because it scores well on Output Sensitivity or Volume Load. Three non-compensatory gate rules are evaluated 
after computing рқ‘Ҷ3 and can only make routing stricter , never more permissive. Gate results are computed by 
prescreen_gate() in sddf/s3_framework.py and cannot be overridden by any weight perturbation. 
вҖў Hard Rule 1 вҖ” HR1 (Knowledge-Critical): if SK = 5 вҶ’ force LLM Only. 
Any workload where SK = 5 is assigned LLM Only regardless of the formula score. The rationale is that SK = 5 
events вҖ” defined as tasks requiring expert-level, licensed, or regulated knowledge вҖ” represent a category of 
epistemic demand that no small model should accept autonomously, regardless of capability estimates. Clinical 
triage (UC6) triggers this rule: patient under-triage is irreversible and legally consequential, and the failure mode 
is not gradual quality degradation but categorical misclassification with life-safety implications. 
вҖў Hard Rule 2 вҖ” HR2 (Max Complexity + High Knowledge): if TC = 5 and SK вүҘ 4 вҶ’ force LLM Only. 
Expert-level generative tasks operating under significant specialized knowledge requirements are disqualified from 
autonomous local deployment. Financial report drafting (UC8) triggers this rule: its formula score of SВі = 3.20 falls 
in the Hybrid range (2.55 вүӨ 3.20 < 3.30), and the formula alone would assign it to the Hybrid tier. However, TC = 5 
combined with SK = 4 overrides the formula, forcing LLM Only. The benchmark confirms that this gate rule is load-
bearing: the best-performing SLM on UC8 achieves only 89.7% parity, a failure rate that would produce systematic 
financial misstatements in production. 
вҖў Flag Rule: if SK вүҘ 4 вҶ’ minimum allowed tier is Hybrid (cannot end as Pure SLM). 
The formula score may push the tier higher but not lower than Hybrid for any workload where SK вүҘ 4. UC1 
(SMS Threat Detection) and UC7 (Legal Contract Risk) trigger this rule. In both cases, the underlying 
specialized knowledge requirement вҖ” threat intelligence interpretation and legal contract analysis, respectively 
вҖ” introduces a category of failure risk that precludes full local autonomy even when the weighted sum score 
might otherwise permit it. 
Together, the three gate rules implement a safety-first architecture: the formula handles the normal operating 
range where compensability is appropriate, while the gate rules handle the extreme cases where safety 
requirements are categorical rather than graduated.

1.4 Fully Worked Example: UC8 (Financial Report Drafting) 
UC8 is a financial report drafting task requiring expert-level generative composition of structured financial narratives 
from raw tabular data. It is the canonical illustration of how the Hard Rule 2 gate overrides a formula score that would 
otherwise underestimate deployment risk. 
Given dimension scores: TC = 5, OS = 1, SK = 4, DS = 3, LT = 2, VL = 1 
 
Step 1 вҖ” Weighted sum computation: 
 
3(5) + 2(1) + 5(4) + 2(3) + 2(2) + 1(1) = 15 + 2 + 20 + 6 + 4 + 1 = 48 
 
Step 2 вҖ” SВі score: 
 
рқ‘Ҷ3new = 48/15=3.2 
 
Step 3 вҖ” Formula tier (before gates): 
Since 2.55 вүӨ 3.20 < 3.30, the formula tier = Hybrid 
 
Step 4 вҖ” Gate check: 
HR2 applies because TC = 5 and SK вүҘ 4 
 
Step 5 вҖ” Final tier: 
HR2 overrides the formula tier вҶ’ final tier = LLM Only 
This example demonstrates the load-bearing nature of HR2: the formula score alone would assign this workload to 
the Hybrid tier, which would permit some degree of local SLM inference. The gate rule recognizes that expert-

small models produce plausible-sounding but factually incorrect outputs вҖ” a risk class that the weighted sum

necessity: the best-performing SLM on UC8 achieves only 89.7% parity, consistent with a systematic failure rate 
that would produce material financial misstatements in a production environment.

The model solves mapping in four steps:

Use the taskвҖҷs prompt_text as the only semantic input. 
2. Zero-shot scoring 
Run MoritzLaurer/deberta-v3-large-zeroshot-v2.0 over label set: 
classification, information_extraction, summarization, text_generation, retrieval_grounded, instruction_following
, maths, code_generation. 
3. Primary + secondary assignment 
Primary family = highest-score label. 
Secondary family is added when score is close to top-1 and prompt constraints require it (citation-only, strict 
schema, explicit quantitative reasoning, or code-fix generation). 
4. Composite decision policy 
If multiple families are active, keep all of them and apply a conservative merge rule downstream: 
disqualified > llm_only > hybrid > pure_slm (strictest family wins). 
This makes mapping reproducible, interpretable, and aligned with how real enterprise prompts mix classification, 
generation, reasoning, and control constraints.

Section 7  Runtime Deployment 
 
How Policy and Empirics Are Connected at Runtime 
Runtime deployment is where the policy artifact (S3) and the empirical artifact (SDDF v3) are jointly enforced. 
The connection is operationally sequential: 
1. S3 defines the enterprise workload as a business use case. 
2. Section 5 maps that use case to its SDDF task family. 
3. SDDF v3 provides the calibrated model artifacts and validated thresholds from the train-validation-test pipeline. 
4. Runtime processes each incoming query through SDDF scoring and threshold comparison. 
5. Query-level routes are aggregated into one use-case level runtime decision. 

===== PAGE 9 =====
 
 
This means S3 determines the governance unit and decision context, while SDDF determines the measurable routing 
behavior under calibrated failure control.

appendix F. Theoretical Properties 
 
F.1 Monotonic Curve Regularization 
build_difficulty_curves() in sddf/validation_dynamic.py applies isotonic regression: capability curve C_m(d) is 
forced monotone non-decreasing in difficulty; risk curve R_m(d) is forced monotone non-increasing. This 
regularizes noisy per-bin empirical estimates before threshold selection. 
 
F.2 Feasibility Guarantee
